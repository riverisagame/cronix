# [SCAN] 幽灵 DAG (依赖引擎) 架构对齐与抉择

## 1. 现状：名存实亡的依赖系统
当前 `dag.go` 提供了完美的 Kahn 算法用于无环图依赖构建与拓扑排序，且 `executor.go` 包含 `buildDAG` 方法用来拉取数据库的 `task_deps`。**但在执行引擎中，没有被任何触发器或调度器调用。**

这意味着：哪怕用户在前端给 "任务B" 设定了 "依赖任务A"，后台仍然是根据各自的 Cron 表达式在跑，一旦 A 耗时变长，B 不会等待 A，导致数据时序全部错乱。

## 2. 需求深挖与架构对齐 (Adversarial Questioning)

为了修复这个问题，存在两种完全不同的系统演进方向。必须进行**硬核审问与对齐**。

### 方案 A：基于任务组的 DAG 工作流 (Group-based Workflow DAG) —— [推荐]
- **实现原理**：在现有的 `TaskGroup`（任务组）模式中（当前只有 `parallel` 和 `sequential`），新增一个 `dag` 模式。
- **触发机制**：整个任务组由统一的 Cron 触发。执行时，抽取组内的所有任务构建局部 DAG。
- **调度逻辑**：根据 `TopologicalSort()` 输出的分层二维数组。第 0 层并行执行 -> 全部成功 -> 第 1 层并行执行 -> ... 如果某层有任务失败，中断后续层。
- **性能对冲**：逻辑极其内聚，无跨度状态管理的开销。依赖局限在业务组内，极大地减少了“环形死锁”的可能。
- **自我攻击**：如果用户想跨组配置依赖，或者某两个全局散装任务配置了依赖，此方案将忽略组外的全局依赖。

### 方案 B：全局事件驱动 (Event-Driven Global DAG) —— [高风险]
- **实现原理**：废弃组调度。每当任意单体任务 `executeTask` 成功完成时，触发一个钩子，去数据库查询 `TaskDep` 看是否有依赖它的下游任务。
- **调度逻辑**：顺藤摸瓜找到下游任务后，还要反向查询该下游任务的所有上游（可能它不仅依赖 A，还依赖 C），然后去 `execution_logs` 查 C 最近是否也成功了。如果都满足，自动触发它。
- **性能对冲**：每次任务结束都要进行重度 DB 联表查询（查询所有上游的状态），在并发高时会拖垮数据库。
- **自我攻击**：状态一致性是极大的灾难。如果任务 C 是昨天的日志成功，A 是今天的成功，下游任务应不应该触发？这需要引入复杂的 "运行批次 (RunID)" 概念，会导致重构幅度极大，违背了当前的极简架构。

## 3. 对现有功能的影响范围评估
- 若采用**方案 A**：
  - 会修改 `RunGroup` 逻辑（增加 `case "dag":`）。
  - 会影响前端/后端的组模式枚举校验（允许传入 `dag`）。
  - 需要在后端确保只有 `Group` 内的任务才能有效利用 `dag`。
  - **副作用风险低**：因为原本 DAG 就没跑起来，这纯粹是新增功能路径。

## 4. 结论与对齐请求
目前的 Cronix 缺乏引入复杂 Airflow 级别全局 Event 引擎的基础设施。强行上**方案B**会导致过度设计与不可控的 Bug（如幻影触发、循环重试风暴）。

**强烈推荐 方案A**。请明确确认是否同意将 DAG 的效力**收缩并落实到 Task Group (任务组)** 的范畴内运行？一旦确认，我将推进 [IR] 计划，彻底激活沉睡的 DAG。
