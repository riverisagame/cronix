# ADR: 原生 SVG 拓扑依赖图与多维优雅退出机制设计

## 1. 背景与上下文
当前项目需要实现以下两个高优先级的功能和逻辑优化：
1. **任务依赖关系 DAG 可视化**：任务依赖非常关键，如果缺乏直观的拓扑图，用户很难理清多级级联任务（A -> B -> C）的链条关系。
2. **优雅退出进程安全**：目前优雅退出简单地使用协程接收信号并在其中执行 `os.Exit(0)` 强行中断，未关闭活动的 HTTP 连接与底层 GORM SQLite 事务及连接，在高并发下容易导致数据写丢失和 SQLite `database is locked` 文件锁死。

## 2. 方案设计与选型

### 2.1. 拓扑图：原生 SVG + 拓扑层级算法 替代 第三方重量级图表库
我们决定不引入如 `D3.js`、`Echarts` 或其他第三方库。

* **理由**：
  1. **零外部体积开销**：Vue + 原生 SVG + 简单的分层排版算法，打包体积增加为零，彻底规避 npm/vite 打包兼容风险。
  2. **完美的工业风格融入**：通过 SVG 滤镜直接在节点上挂载我们的毛玻璃 CSS 类名，并让连线拥有激光霓虹蓝/绿的高光动画，与“极简工业控制台”风格融为一体。
  3. **拓扑分层布局 (Layered Graph Layout)**：在 JavaScript 中，查出所有任务依赖后，通过计算入度，递归归纳每个节点所在的层级（Level 0, Level 1...），并据此计算 $X, Y$ 坐标并渲染连接曲线。

### 2.2. 全量预载接口设计
* **问题**：如果在前端绘制全局拓扑图，需要查出所有任务的依赖。如果每个任务单独发 `/api/tasks/:id/deps` 接口，在任务多时会瞬间产生 $O(N)$ 个并发请求，会直接触发系统已有的 `429 Rate Limiter` 限流封禁。
* **选型**：在 `model.Task` 中添加 `DependsOnIDs []uint` (`gorm:"-" json:"depends_on_ids,omitempty"`) 非持久化字段，并在 `ListTasks` 列表查询中批量预加载填充。前端调用一次 `taskAPI.list({page:1, page_size:1000})` 即可直接拉回全部的拓扑数据，将网络交互降低至 $O(1)$。

### 2.3. 系统多维优雅退出机制 (Graceful Shutdown)
我们在主协程中控制退出链条，替换原先的 `os.Exit`：
1. 捕获中断信号后，主协程率先调用 `http.Server.Shutdown(ctx)` 优雅切断 HTTP 流量（拒绝新连接，处理完已有请求）。
2. 调用 `engine.Stop()` 关闭定时触发。
3. 调用 `exec.Shutdown()` 等待线程池 `ants` 中已在运行的作业清空。
4. 获取底层 `database.DB` 的连接实例，调用 `Close()` 安全断开 SQLite 物理连接，彻底消除写冲突和死锁残余。

## 3. 并发安全与性能对冲 (Performance & Safety)
* **并发释放**：优雅退出限时 5 秒（使用 `context.WithTimeout`）。如果 5 秒内线程池未释放完成，将强制终止进程，防止出现长挂起僵尸进程。
* **物理零污染**：修改仅涉及退出的信号释放管道与依赖挂载字段，对现有的数据库实体表结构无任何破坏性侵入。
