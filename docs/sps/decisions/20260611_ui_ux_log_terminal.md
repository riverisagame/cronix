# UI/UX 优化决策: 流式日志与进程控制面板

## 1. 需求分析与痛点识别
- **当前状态**: `TaskList.vue` 右侧抽屉展示历史执行记录，无实时执行状态流，点击"Run"仅一闪而过的提示，缺乏深层交互。
- **痛点**: 大段日志滚动困难、报错日志难以检索、长进程无法及时打断，缺乏类似 GitHub Actions 的专业流水线体验。

## 2. 方案选型 (基于 ui-ux-pro-max)
- **Pattern**: Data-Dense + Drill-Down (数据密集型钻取)
- **Typography**: `Fira Code` (等宽字体，专业代码与日志展示) + `Fira Sans` (主体阅读)。
- **交互规范**:
  1. **悬浮操作台 (Sticky Action Bar)**: 在日志顶端固定显示进程当前状态与红色 `Kill Process` 强杀按钮，避免滚动后找不到终止开关。
  2. **搜索高亮**: 日志框内建即时搜索框（Ctrl+F拦截或原生输入），高亮匹配关键字。
  3. **智能滚动控制 (Auto-scroll)**: 默认锁定底部 (`scroll-behavior: smooth`)，一旦用户手动上划，取消自动锁定并展示 "Resume scrolling" 悬浮按钮。
  4. **沉浸式暗黑终端**: 日志背景 `#1E293B` (slate-800)，文本采用高对比色，以降低高频闪烁对视力的刺激。

## 3. 对现有功能的影响
- **UI 组件层**: 仅扩展 `TaskList.vue` 的 Drawer 内容逻辑，并新增轮询 API，**无破坏性影响**。
- **并发与性能**: 日志请求仅为读取本地磁盘文件 `GET /api/tasks/:id/stream`，规避了 SQLite 并发锁死。
- **组件结构**: 拟将抽屉拆分为两个 Tab："Live Console" (实时) 与 "Execution History" (历史)。

## 4. 遗留问题与风险
- **风险**: 若日志文件极大（如数百MB），频繁全量拉取 `/stream` 会导致带宽与前端内存崩溃。
- **对冲策略**: 前端采用文本截断 (如只展示最后 10000 行) 或后端实现 `Range` 局部流传输？由于目前只是文件读取，我们需要限制前端渲染节点的数量（即 DOM 虚拟化，或者仅仅利用 `textarea` / 原生 `pre` 防止卡顿）。
