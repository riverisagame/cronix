# Notification System Hardening ADR

## 1. 现状痛点 (The Problem)
当前系统中，任务执行完毕后的通知投递机制存在严重瓶颈，具体表现为：
1. **消费者单点阻塞**：在 `internal/notify/notifier.go` 中，`Start` 方法仅使用了一个**单线程（单 Goroutine）**串行消费通知。
2. **长尾耗时拖垮吞吐量**：`sendWebhook` 涉及 HTTP 网络请求，且带有 `retry` 和 `time.Sleep`。如果对端接口响应慢或者失败，单次投递可能耗时 5~15 秒。这期间整个消费者被锁死，无法处理其他通知。
3. **静默丢包现象**：缓冲通道 `notifyCh` 容量硬编码为 256。由于单消费者处理过慢，一旦遇到并发任务洪峰（例如上百个任务在同一时刻执行完毕），队列瞬间爆满。此时 `internal/scheduler/executor.go` 的投递逻辑（`select + default`）会直接**丢弃事件并打印 Warn 日志**。用户无法收到警报，引发安全感崩塌。

## 2. 需求深挖与质疑 (Adversarial Questioning)

### Q1: 丢弃逻辑的初衷是什么？能否去掉 `default` 让它阻塞？
- **实现审计**：不能去掉。如果发送端（Executor 的调度线程）被 `notifyCh <- event` 阻塞，它将无法完成任务清理和回收，引发系统级雪崩。
- **自我攻击**：用阻塞代替丢弃是灾难性的。Executor 池将被挂起，调度器彻底停摆。

### Q2: 能否简单增加 Channel 缓冲区大小？
- **实现审计**：将 256 增加到 10000。
- **自我攻击**：治标不治本。如果对端持续不通（Webhook 宕机挂起 1 分钟），再大的内存也会被打满。此外，如果进程重启，内存中的队列会全部丢失。

### Q3: 最稳健的改造方案应该是什么？
- **性能对冲**：通知的核心诉求是“高并发下的非阻塞”与“高成功率”。我们既不能阻塞核心调度器，又不能让通知单线程排队。

## 3. 架构选型 (Architecture Options)

### 方案 A：异步 Worker 池 + 缓冲 (In-Memory Worker Pool)
为 `Notifier` 引入并发控制的 Goroutine 协程池。
- **工作流**：主循环仍然从 `notifyCh` 取数据，但取到后放入 `ants` 协程池（或自己控制并行的 goroutine）并发投递。
- **优点**：改动极小。假如开 50 个 Worker，原本 15秒/条 的处理速度变为 15秒/50条，吞吐量提升 50 倍，有效抵抗洪峰。
- **缺点**：仍是内存队列，进程崩溃时未发出的通知仍会丢失；队列满时仍需降级丢弃。

### 方案 B：落盘与异步调度队列 (Persistent Background Queue)
引入标准的异步任务中间件（例如利用现有的 DB 表新建一个 `notifications` 持久化表），或者使用类似于 Machinary 的方案。Executor 仅向 DB 写入“待通知”记录。独立后台协程定期拉取 DB 进行投递。
- **优点**：数据绝对不丢（Zero Data Loss），抗服务重启。
- **缺点**：引入极大的系统复杂度，增加数据库 I/O 负担，需要维护通知重试表、状态、清理逻辑等。

## 4. 推荐意见
考虑到 `Cronix` 的系统定位为轻量级调度系统，**我们推荐【方案 A（异步 Worker 池）】为主，辅以【非阻塞投递机制的优化】。** 
对于一个轻量系统，Webhook 网络层面的并发处理往往能解决 99% 的拥堵问题，暂时无需引入重型的数据库落盘队列方案。
