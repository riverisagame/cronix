## 📖 第一部分：先看热闹 —— 这玩意儿是干啥的？

### 🧠 一句话概括

**Cronix = 一个会看课程表的智能闹钟管家。**

想象一下：
- 你妈妈每天半夜要起床备份电脑文件，很辛苦
- 你希望有个机器人，到了晚上 12 点自动帮你备份，不用人管
- 你想知道它有没有干活、有没有出错

Cronix 就是这个机器人！它能：
1. **定时干活**：到了你设定的时间，自动执行命令
2. **多种任务**：可以执行脚本、访问网页、清理垃圾文件、检查服务健康
3. **查看日志**：随时看你设置的定时任务运行成功了没
4. **画图展示**：有一个漂亮的网页界面，像仪表盘一样展示所有数据

### 🎮 三个生活化场景

| 场景 | 类比 | Cronix 怎么做 |
|------|------|--------------|
| 🗄️ **定时备份** | 每天放学自动整理书包 | 每天凌晨 3 点，自动备份数据库 |
| 🌐 **网页监控** | 门卫大叔每小时巡逻 | 每 5 分钟检查你的网站还能不能访问 |
| 🧹 **清理垃圾** | 周末大扫除 | 每周日清空临时文件夹 |

### 🏗️ 技术栈速览（看不懂没关系，后面会解释）

| 层级 | 用了什么 | 一句话 |
|------|---------|--------|
| 后端 | Go 语言 + Gin 框架 | 像餐厅后厨，负责干活 |
| 前端 | Vue 3 + TypeScript | 像餐厅前台，好看又好用 |
| 数据库 | SQLite | 像记账本，记录所有事情 |
| 部署 | Docker | 像搬家箱，一键打包带走 |

┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐

### 🟡 Level 2：初级冒险者 —— 从入口看起

> **目标**：看懂程序是怎么启动的，数据长什么样

#### 第 3 站：`main.go` —— 程序的大门 🚪

打开 `main.go`，你会看到非常详细的注释。这是全项目注释写得最好的文件！

**关键代码（只有 3 行是真正干活的）**：

```go
package main          // ← "我是独立程序，不是零件"

func main() {          // ← "开始！"
    if err := cmd.Execute(); err != nil {  // ← 把工作交给 cmd 包
        os.Exit(1)     // ← 出错了就退出
    }
}
```

**💡 初学者收获**：
- 每个 Go 程序都有一个 `main()` 函数，它是"起跑线"
- `main.go` 自己不干活，它是个"老板"，把工作分派给 `cmd/` 包
- `os.Exit(0)` = 正常下班，`os.Exit(1)` = 出错撤退

#### 第 4 站：`cmd/root.go` —— 启动总指挥 🎬

这个文件是整个程序的"总指挥"，管着所有子命令。想象它是一个遥控器：

```
┌──────────────────────────────────┐
│         cronix 遥控器             │
│                                  │
│  [▶ serve]  启动服务器            │
│  [🔑 passwd] 改密码               │
│  [📋 logs]   看日志               │
│                                  │
│  你输入 "cronix serve"            │
│  → Cobra 库识别 "serve"           │
│  → 执行 runServe() 函数           │
│  → 按顺序：加载配置→初始化数据库    │
│    →启动调度→启动HTTP服务→等待信号  │
└──────────────────────────────────┘
```

**💡 初学者收获**：
- `init()` 函数在 `main()` 之前自动运行，用来做准备工作
- `defer` 是 Go 的"无论如何最后都要做"的承诺（比如关数据库连接）
- `runServe()` 按固定顺序启动各个模块，像开机自检一样

#### 第 5 站：`internal/domain/model/task.go` —— 任务长什么样 📝

这个文件定义了"任务"的数据结构。打开看看，你会发现一个叫 `Task` 的 `struct`（结构体），它就像一个表格的设计图：

| 字段名 | 类型 | 含义 | 比喻 |
|--------|------|------|------|
| `ID` | 数字 | 唯一编号 | 学号 |
| `Name` | 文字 | 任务名称 | 课程名 |
| `CronExpr` | 文字 | 定时表达式 | 课程表上的时间 |
| `TaskType` | 文字 | 类型 (shell/http/...) | 语文课/数学课 |
| `Command` | 文字 | 要执行的命令 | 课堂上要做的题 |
| `Enabled` | 布尔 | 是否启用 | 这节课还上不上 |

```go
type Task struct {
    ID        uint      `gorm:"primaryKey" json:"id"`
    Name      string    `gorm:"not null" json:"name"`
    CronExpr  string    `json:"cron_expr"`
    TaskType  string    `gorm:"not null" json:"task_type"`
    // ... 还有更多字段
}
```

**💡 初学者收获**：
- `struct` = 表格设计图，"字段" = 表格的列
- 反引号 \`gorm:"..." json:"..."\` 是"标签"，告诉工具怎么处理这个字段
- GORM 会读这个结构体，自动在数据库里建表——你不用写 SQL

#### 第 6 站：`web/src/router/index.ts` —— 网页路线图 🧭

这个文件定义了浏览器地址栏到页面的对应关系。就像一个导航系统：

| 你访问的网址 | 显示的页面 | 需要登录吗 |
|-------------|-----------|-----------|
| `/login` | 登录页 | ❌ 不用 |
| `/` | 仪表盘首页 | ✅ 需要 |
| `/tasks` | 任务列表 | ✅ 需要 |
| `/tasks/new` | 新建任务 | ✅ 需要 |
| `/logs` | 执行日志 | ✅ 需要 |
| `/settings` | 设置页 | ✅ 需要 |
| `/groups` | 任务组 | ✅ 需要 |

```typescript
// 每条路由规则 = 一个网址对应一个页面
{ path: '/tasks', component: () => import('../views/TaskList.vue') }
//   ↑ 网址            ↑ 懒加载：打开时才下载这个页面文件
```

**💡 初学者收获**：
- 前端路由 = "网址 → 页面"的对照表
- `() => import(...)` 是"懒加载"：用到才下载，首次打开更快
- `requiresAuth: true` = 这页需要登录才能看（门禁）

**🎉 恭喜你升到初级！** 你现在知道程序怎么启动、数据怎么存、网页怎么跳转了。

### 🟠 Level 3：中级骑士 —— 核心逻辑

> **目标**：看懂任务是怎么被定时触发和执行的

#### 第 7 站：`internal/application/scheduler/engine.go` —— 心脏起搏器 ⏰

这是整个项目最核心的文件之一。它像一个永远不会停的钟表：

```
┌─────────────────────────────────────────┐
│           调度引擎工作流程               │
│                                         │
│  1. 从数据库加载所有"启用"的任务          │
│  2. 每个任务解析它的 Cron 表达式          │
│  3. 把任务注册到 robfig/cron 库          │
│  4. robfig/cron 内部维护一个"最小堆"      │
│     （把下次触发时间排序，只看最近的）     │
│  5. 到了时间 → 把任务ID推进 channel       │
│  6. 循环回到第 4 步                      │
│                                         │
│  调度引擎  ──(任务ID)──→  triggerCh     │
│  (只看表)      channel     (传送带)      │
└─────────────────────────────────────────┘
```

**💡 初学者收获**：
- "调度"和"执行"是分开的——调度只管"到点了没"，执行才管"怎么干"
- `channel` 是 Go 的传送带：一头放东西，另一头取东西，双方不用见面
- `robfig/cron` 库用"最小堆"数据结构管理海量定时任务，效率很高

#### 第 8 站：`internal/interfaces/handler/task.go` —— 服务员 🧑‍🍳

这个文件处理所有来自网页的请求。想象你在一家餐厅：

```
你(浏览器)  →  "我要看任务列表"  →  handler  →  service  →  database
            ←  JSON 数据回来      ←          ←           ←
```

```go
// 一个典型的 Handler 长这样：
func (h *TaskHandler) ListTasks(c *gin.Context) {
    tasks, err := h.TaskSvc.List()   // 叫 service 去查数据
    if err != nil {
        respondError(c, ...)         // 出错了就报错
        return
    }
    respondOK(c, tasks)              // 成功就返回数据
}
```

**💡 初学者收获**：
- Handler = 服务员：接收请求 → 调用服务 → 返回结果
- `c *gin.Context` 是贯穿每个请求的"神奇包裹"，装着请求的所有信息
- 输入验证很重要！永远不要相信用户发来的数据

#### 第 9 站：`internal/interfaces/router/router.go` —— URL 总地图 🗺️

这个文件把所有 URL 路径和处理函数连起来：

```
GET    /api/health        → 健康检查（不用登录）
POST   /api/login         → 登录（不用登录）
GET    /api/tasks         → 列出所有任务（需要登录）
POST   /api/tasks         → 创建新任务（需要登录）
GET    /api/tasks/:id     → 查看某个任务（需要登录）
PUT    /api/tasks/:id     → 修改某个任务（需要登录）
DELETE /api/tasks/:id     → 删除某个任务（需要登录）
POST   /api/tasks/:id/run → 手动运行任务（需要登录）
GET    /api/dashboard/stats → 仪表盘统计（需要登录）
```

**💡 初学者收获**：
- RESTful API：用 HTTP 动词(GET/POST/PUT/DELETE)表达"查/增/改/删"
- `:id` 是动态参数，`/api/tasks/5` 和 `/api/tasks/99` 走同一个处理函数
- 大部分接口需要 JWT 认证（你得先登录拿"门票"）

**🎉 恭喜你升到中级！** 你现在知道调度引擎怎么工作、HTTP 请求怎么处理了。

### 🔴 Level 4：高手之路 —— 深入细节

> **目标**：理解执行器、任务依赖和基础设施

#### 第 10 站：`internal/application/scheduler/executor.go` —— 执行工人 👷

调度引擎只管"到点了"，而执行器才是真正干活的人：

```
triggerCh (传送带)
    │
    ▼
┌──────────────────────┐
│   执行器 (Executor)   │
│                      │
│  用 goroutine 池     │
│  最多同时跑 32 个任务  │     ← pool_size: 32
│                      │
│  每个任务独立运行      │
│  超过 1 小时强制停止   │     ← max_timeout_sec: 3600
│  内存不超过 512MB    │     ← memory_limit_mb: 512
└──────────────────────┘
    │
    ▼
  执行结果 → 写入日志
```

**💡 初学者收获**：
- `goroutine` 是 Go 的"轻量级线程"——可以同时跑几千个，几乎不占内存
- `ants` 是一个 goroutine 池库：控制最多同时跑几个，防止系统被压垮
- 执行器有安全阀：超时会被杀掉，内存超限会被干掉——防止一个任务拖垮整个系统

#### 第 11 站：`internal/application/scheduler/dag.go` —— 任务间的"先后顺序" 🔗

有些任务有依赖关系：A 完成了，B 才能开始。这就像一个"有向无环图"（DAG）：

```
         任务A (备份数据)
           │
           ▼
         任务B (压缩文件)  ← 等 A 完成后才能跑
           │
      ┌────┴────┐
      ▼         ▼
   任务C      任务D       ← C 和 D 可以同时跑，都等 B 完成
  (上传云)   (发通知)
```

**💡 初学者收获**：
- DAG = 有向无环图，用来表达"谁先谁后"
- 不能有环（A→B→C→A），否则会死循环——代码会检测并拒绝
- 这是 LeetCode 里经典的"拓扑排序"问题的实际应用

#### 第 12 站：`internal/infrastructure/` 层速览 🔧

| 文件 | 做什么 | 比喻 |
|------|--------|------|
| `config/config.go` | 加载 config.yaml | 读说明书 |
| `database/database.go` | 连接数据库 | 打开记账本 |
| `cache/cache.go` | 内存缓存 | 便签纸（记在脑子里更快） |
| `circuit/circuit.go` | 断路器 | 保险丝（出错太多就暂停，防止烧坏） |
| `notify/notifier.go` | 发送通知 | 邮递员（任务出错时发邮件/webhook 通知你） |

## ⛽ 第四部分：概念加油站 —— 术语一网打尽

这一部分用最直白的比喻解释你会在代码里遇到的术语。

| 术语 | 一句话 | 生活比喻 |
|------|--------|---------|
| **Cron 表达式** | 用 5 个数字描述"什么时候执行" | `0 8 * * 1` = "每周一早上 8 点"，就像课程表 |
| **Goroutine** | Go 的轻量级"同时做事"的方式 | 像开了很多个窗口同时写作业，互不影响 |
| **Channel** | goroutine 之间传数据的管道 | 工厂的传送带：一头放、一头取、中间不碰面 |
| **REST API** | 用 HTTP 动词操作数据的规范 | 餐厅：GET=看菜单、POST=下单、PUT=换菜、DELETE=取消 |
| **JWT** | 登录后的"门票" | 游乐园手环：进门时戴上，玩每个项目时亮一下就行 |
| **DDD 分层** | 代码按职责分楼层 | 公司：老板(入口)→部门经理(业务)→基层员工(执行)→仓库(数据) |
| **ORM** | 把数据库表自动翻译成代码对象 | 翻译官：你说"查所有任务"，它翻译成 `SELECT * FROM tasks` |
| **Docker** | 把程序和它的"家"打包带走 | 搬家集装箱：不管搬到哪台电脑，打开就能用 |
| **Middleware** | 请求到达前的"安检站" | 进地铁站要刷卡——中间件就是那个闸机 |
| **Circuit Breaker** | 出错太多就暂停，过会儿再试 | 保险丝：电流太大就熔断，保护电器 |

### 🔢 Cron 表达式详解

这是你在代码里最常见的"密码"之一。它有 5 个数字，空格隔开：

```
  *    *    *    *    *
  │    │    │    │    │
  │    │    │    │    └── 星期几 (0-6, 0=周日)
  │    │    │    └────── 月份 (1-12)
  │    │    └─────────── 日期 (1-31)
  │    └──────────────── 小时 (0-23)
  └───────────────────── 分钟 (0-59)

例子：
  */5 * * * *    → 每 5 分钟一次
  0 8 * * 1      → 每周一早上 8:00
  0 2 * * *      → 每天凌晨 2:00
  30 18 1 * *    → 每月 1 号下午 6:30
```

2. 点"新建任务"
3. 填一个简单的：
   - 名称：`测试任务`
   - Cron 表达式：`*/5 * * * *`（每 5 分钟一次）

### 实验 4：跟着请求走一遍

打开浏览器的开发者工具（F12 → Network 标签），然后：

1. 刷新页面 —— 看发起了哪些请求
2. 点进一个请求 —— 看 URL、请求方法、返回的 JSON 数据
3. 回到代码，找到对应的 Handler 函数 —— 看它是怎么处理这个请求的

**这条链路是**：浏览器 → HTTP 请求 → Gin 路由 → Handler → Service → 数据库 → 返回 JSON → 浏览器渲染
